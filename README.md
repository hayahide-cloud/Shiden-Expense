# Shiden Expense（旧: ExpenseWeb）

`ExpenseVBA`（Excel VBA版・経費精算システム）のモバイル用コンパニオンWebアプリ。

スマホで領収書を撮影 → OpenAI Vision APIでOCR取込 → 明細として記録する。
ExpenseVBAを置き換えるものではなく**併用**する。月次集計・仕訳出力は引き続きExcel側（ExpenseVBA）で行う。

本リポジトリには2つの実装がある。**現在の運用はサーバーレス版（Shiden Expense）**。

| | Shiden Expense（`docs/`） | フル構成（`frontend/` + `backend/`） |
|---|---|---|
| 状態 | **運用中** | 実装済み・デプロイ保留（将来のセルフホスト移行先） |
| ホスティング | GitHub Pages（無料） | Railway or 自宅サーバー |
| サーバー | 不要（ブラウザから直接OpenAI API呼び出し） | FastAPI + PostgreSQL |
| データ保存先 | 端末内（localStorage） | PostgreSQL |
| コスト | OpenAI API利用料のみ | ＋ホスティング費 |

## Shiden Expense（サーバーレス版）

`docs/index.html` 1ファイルで完結する静的Webアプリ。紫基調のダークテーマ。

- 取込タブは**「購入費・立替費の精算」「交通費の精算」「宿泊費の精算」の3つの大項目**に
  分かれる。購入費側は「タップして撮影 / 画像を選択」と「手入力」、交通費側は
  「経路のスクリーンショットを選択」（＋PC限定の貼り付け・画面キャプチャ）と
  「🎫 タップして撮影 / 画像を選択」（タクシー・飛行機等の紙のレシート/領収書用。
  トリミング→レシートOCRの通常フローで、勘定科目だけ交通費に固定）と
  「手入力」、宿泊費側は購入費と同じ撮影＋手入力。
  交通費の証憑は「経路スクショ」「レシート」の2種類があるため、保存時に種別を
  `img_kind`（route/receipt）として記録し、再解析ではこの値でプロンプトを自動選択する
  （全PNG化以前の旧データは保存形式PNG/JPEGで判別）。
  写真の後付けでは選択ファイルの形式で判別する（スクショ=PNG・カメラ写真=JPEG/HEICという
  OSの性質を利用）。
  交通費の手入力は勘定科目=交通費・日付=今日を初期値にフォームが開く。
  宿泊費の入口から取り込んだ明細は、OCRの判定によらず勘定科目が「宿泊費」に固定される
  （手入力も宿泊費で開く。入口が種別を決める。トリミング中のリロードにも耐える）。
  大項目は一覧と同じ配色で色分けされる（購入費・立替費=アンバー、交通費=ティール、
  宿泊費=ローズ。見出し・選択ボタン・手入力ボタンに適用）。
  ※勘定科目に「宿泊費」を追加したため、ExpenseVBA（Excel側）の科目リストにも
  「宿泊費」の追加が必要
- **往復区分**: 勘定科目が交通費のときだけ「往復区分（任意）」欄が表示され、
  往路／復路／往復を選べる。一覧の科目行に「交通費（往復）」のように表示され、
  申請文に「往復区分: 往復」の行が入り、CSVは末尾に「往復区分」列を追加
  （交通費以外は空欄）。「往復」を選ぶと片道運賃を2倍にし、計算式を
  「往復: ¥210 × 2 = ¥420」の形で摘要に1行残す（自動計算を見える形で明示）。
  往復をやめるとその行から片道運賃を読み取って金額を戻し、行も消す。
  金額未入力で往復を選んだ場合は何もしない（金額を入れてから選び直せば計算される）。
  さらに「往復」を選ぶと**「復路の日付（任意）」欄**が現れ、往路と復路が別日の場合に
  期間として記入できる（一覧は「往路日 〜 復路日」表示、申請文の日付も期間になる。
  宿泊費の宿泊終了日と同じ`date_to`・CSVの「終了日」列を共用）
- **宿泊期間**: 勘定科目が宿泊費のときだけ「宿泊終了日（任意・チェックアウト日）」欄が
  表示され、複数日にわたる宿泊を期間で記入できる（日付=宿泊開始日）。
  開始日より前の終了日はエラーで保存できない。一覧は「開始日 〜 終了日」の期間表示、
  申請文の日付も「開始日〜終了日」になる。CSVは末尾に「終了日」列を追加
  （宿泊終了日または交通費（往復）の復路日付が入る。それ以外は空欄。
  領収書日付列は開始日のまま＝Excel側の日付処理を壊さない）。
  宿泊費の撮影取込では、領収書に宿泊期間の記載があればOCRがチェックイン日／
  チェックアウト日を自動で読み取る
- 撮影 → その場でOCR（OpenAI Vision API直呼び） → 編集フォーム → 端末内に保存
- 証憑画像も端末内（IndexedDB）に保存。撮影レシートは長辺1800pxに自動縮小
- **証憑は全PNGで保存し、明細CSVを画像内に埋め込む**: すべての証憑（撮影レシート・
  経路スクショ・PDF）をPNGで統一し、PNGのiTXtチャンク（公式のテキスト領域、
  キーワード`ShidenExpense`）に**その明細1行分のCSVを埋め込む**。画素には触れない
  ため画質への影響はゼロ。ファイル保存・ZIP・メール添付ではそのまま保持されるので、
  「証憑画像さえ残っていれば明細を復元できる」保険になる（LINE・SNSへの投稿は
  再エンコードで消える）。明細を編集して保存すると埋め込みCSVも更新される。
  チャンクのCRC32はZIP自前実装のものを共用（追加ライブラリなし）。
  読み出し用の`extractCsvFromPng()`も実装済み（将来の「画像から明細復元」の入口）。
  ※撮影レシートはJPEG比で容量が数倍になるトレードオフを承知の上でのオーナー決定。
  交通費の証憑種別（経路スクショ/レシート）は形式でなく`img_kind`フィールドで
  記録・判別する（全PNG化に伴い、旧来のPNG/JPEG判別から移行。旧データは形式で判別）
- **手動トリミング（iOSの「書類をスキャン」風・全画面の編集フェーズ）**: 写真を選択すると
  ヘッダー・下部ナビを隠した全画面の編集画面に切り替わる。画像は画面内に完全に収まる
  サイズで表示されるためスクロールが存在せず、「解析する」「キャンセル」ボタンと
  補正のON/OFFは常に画面下部に見えている。画像の上に四隅の
  ドラッグハンドルを表示し、領収書の角に合わせて切り抜ける。初期位置は常に
  横15%・縦10%内側の固定位置（以前は背景との輝度差による自動検出を行っていたが、
  実写では明るい机・床を紙と誤検出することが多く、毎回同じ位置に出る予測可能な
  動作の方が合わせやすいため廃止）。切り抜き範囲の確定は人間が行うため、
  金額等が欠落する事故が起きない。
  傾けて撮った紙も解析時に射影変換（自前実装・逆写像＋バイリニア補間）でまっすぐに
  補正される。ドラッグ中は指で隠れる角の周辺を拡大鏡（約2.5倍・十字線つき）で表示する。
  さらに影消去（ブロック最大輝度による照明マップ推定→白基準の正規化。スキャナーアプリの
  「影消去」と同じ原理）と、明るさ・コントラストの自動補正（オートレベル）を
  ONにすると、影のかかった紙も均一な白にしてからOCR・保存に使える
  （どちらも既定はOFF＝元の写真のまま）。補正のON/OFFチェックを切り替えると
  編集中の画像プレビューへ即時反映され、その場で結果を確認できる（実際の保存・OCR用の
  補正は切り抜き後の高解像度画像に対して行う）。切り抜きは高解像度の元画像（長辺3600px）に対して
  行い、保存・送信用の1800px上限は切り抜き後に適用するため、切り抜いても粗くならない
- **交通費の取込（経路スクショ）**: Googleマップ等の経路検索アプリのスクリーンショットを
  選ぶと、運賃（IC運賃優先）・区間・路線名をOCRで読み取り、
  勘定科目「交通費」の明細として確認・修正フォームに反映する。行程（区間ごとの内訳）は
  「出発地点→到着地点」の1行1区間で読み取り、摘要欄に自動入力する。
  電車・新幹線・バス等の**交通機関の区間のみ**を対象にし、徒歩の区間は含めない
  （精算対象にならないため。プロンプトで除外を指示）。
  行程が読めなかった場合は全体区間「出発地→到着地」を入れる。
  摘要の末尾に**移動距離（概算）**を「距離: 約894km」の形式で自動記載する。
  乗換スクショには電車区間の距離が写らないため、画像内の記載があればそれを優先し、
  無ければAIが営業キロの一般知識から概算する（±10%目安・「約」付き。
  自信が持てない場合は0が返り、行ごと省かれる。外部の経路APIは使わない＝
  サーバレス構成のまま。不要なら摘要から行を消すだけ）。依頼内容・用途は
  空のまま残し、出張の用件等を書く欄として使う。
  ※摘要は文字数制限なし・改行可の複数行欄（以前はExpenseVBAに合わせ20文字上限だったが、
  行程を入れる用途のため撤廃）。スクショは紙と違い
  傾き・影が無いため、トリミング編集をスキップして直接解析する。
  Googleマップのスクショには時刻はあっても日付が無いため、読み取れない場合は
  **空のまま**にする（勝手に今日を入れない。空なら要確認バナーで入力を促す）。
  スクショ自体は証憑画像として保存される
  - **分割スクショの複数枚選択に対応**（最大8枚・上から順に選択）: スクロール
    スクリーンショットが撮れない画面向け。OCRには全枚をそのまま個別に渡して
    読み取り精度を保ち（1枚に縮小結合すると縦長になりすぎて文字が潰れるため）、
    証憑画像としては縦に連結した1枚を生成・保存する
  - 連結の出力は幅1920pxまで原寸のまま（iPhoneスクショ幅1290pxは無縮小）・
    **PNG（無劣化）で保存**する。文字中心のスクショは原本性を優先するオーナー判断
    （容量はJPEGの3〜5倍。メール添付上限・表示メモリへの影響は許容）。
    ※その後、明細CSV埋め込みのため撮影レシート等も含め証憑は全PNGに統一した
  - 連結時は連続する2枚の**重なりをピクセル照合で自動検出**し、重複部分を重ねて
    自然に繋ぐ（2枚目以降の上端のステータスバーも継ぎ目で消える）。照合は
    粗探索（幅200px縮小版）→実解像度での精密照合→重なり範囲全体の検証の3段階＋
    浅い重なり用の後向き照合。確信が持てない継ぎ目は、共通の固定UI（ステータスバー・
    下部バー）だけ取り除いて縦に並べ、warnメッセージで知らせる。
    採用した継ぎ目は、重なり範囲内で最も「静かな」行（余白など、横方向の輝度変化が
    少ない行）まで下げてから切り替える（`quietSeamShift`）。照合が数pxずれていても
    文字行の途中で切り替わって崩れて見えるのを防ぐ（ずらしても総高さは変わらない）
  - 取込は2段階: 選択するとまず**連結結果の確認ステップ**が表示され（タップで拡大可）、
    「解析する」を押して初めてOCRが実行される。連結に問題があった場合に
    API呼び出しを無駄にしないため
  - **クリップボード貼り付けにも対応**（主にPC向け）: コピーした画像をCtrl+V（⌘V）で
    取り込める。確認ステップ表示中の貼り付けは「次のページ」として追加連結されるため、
    分割スクショを1枚ずつコピペで積んでいける。取込タブ表示中のみ反応し、
    トリミング中・明細編集中は反応しない。ヒント表示はPC（ホバー可能な端末）のみ
  - **画面キャプチャでの取込（PC限定）**: `getDisplayMedia`で別タブ（Googleマップ等）の
    映像を共有し、経路を**ゆっくりスクロールするだけ**で自動でページを取り込む
    （フレーム到着駆動の`MediaStreamTrackProcessor`で約0.25秒間隔にサンプリングし、
    「前ページから内容が変わった」フレームを取り込む。**アプリのタブが裏に回っていても
    取込が続く**＝マップのタブでスクロールしている間も動く。タイマー＋video要素方式は
    裏タブでタイマー抑制・映像更新停止により取込が止まるため使わない（非対応ブラウザのみ
    フォールバック）。スクロール中のフレームも対象、同じ画面は重複取込しない。
    キャプチャ解像度が途中で変わった場合は寸法の違うページを混ぜず取り直す。
    最大50ページ、手動追加ボタンも補助として残す）。
    取込ページは連結出力の幅上限と同じ1920px幅に取込時点で縮小する
    （最終品質は変わらずメモリを抑えられ、裏タブ低解像度→表で高解像度化の
    解像度の揺れも縮小後は同寸に揃って取り直しが減る）。
    変化判定の閾値も小さめ（感度高め）にして少しのスクロールでもページを重ね、
    ページ間の重なりを大きくして継ぎ目の照合失敗（内容の二重化）を起きにくくする。
    「連結して確認へ」を押した瞬間に、今見えている画面を最後のページとして必ず
    取り込む（最後のひとスクロールが小さいと変化判定に満たず、ページ最下部が
    欠けたまま連結されるのを防ぐ。前ページと完全に同じ画面なら追加しない）。
    映像フレームはスクショと違いピクセルが完全一致しない（OSの表示スケーリング等）ため、
    キャプチャ由来の連結は照合閾値を2.5倍に緩めて繋ぐ。OCRへは取り込んだ生ページではなく、
    重複除去済みの連結画像を1400px単位で分割したもの（最大12枚）を送る。
    連結キャンバスの面積上限はiOSのみ14Mpx（Safariの制限対策）、PCでは48Mpxまで
    許容し、長い経路でも縮小せず原寸を保つ。
    画質対策: `getDisplayMedia`にideal 4Kの解像度制約を渡してネイティブ解像度で
    取り込む（既定はOSスケーリング後の論理解像度でぼやける）。
    さらにCaptured Surface Control対応ブラウザ（Chrome 136+）では
    「🔍 共有タブを150%に拡大」ボタンを表示し、共有中のタブを段階ズーム
    （`increaseZoomLevel`）で150%まで拡大＋プレビュー上のホイール操作を共有タブへ
    転送（`forwardWheel`）する＝タブを切り替えずにスクロールして取り込める。
    ブラウザの仕様上、これらのAPIはユーザーのクリック（transient activation）を
    起点にしないと呼べないため「タブ選択の瞬間に自動で拡大」は不可能で、ボタン方式を
    採用（初回はブラウザの許可プロンプトが出る）。
    あわせてConditional Focus（`CaptureController.setFocusBehavior("no-focus-change")`、
    Chrome 109+）で、共有開始時に選択したタブへ自動で切り替わらないようにしている
    ＝アプリに留まったまま拡大ボタンを押し、自分のタイミングで共有タブへ移動できる。同APIに captured tabの
    「スクロール位置を先頭へ移動」のような直接命令は存在しない。
    拡大すると映像の縮尺が変わり拡大前のページと繋がらないため、
    `zoomlevelchange`で取込済みページを破棄して取り込み直す。
    フレームは重なり連結エンジンで1枚に繋がるため、スクロールスクリーンショット相当の
    証憑画像が作れる。ウィンドウ／画面全体の共有が選ばれた場合は、アプリ自身の画面が
    混入するためタブ共有を促すwarnを出す。iOS Safariは同APIに非対応のためボタン自体をPCでのみ表示。
    連結の重なり判定は「重なり範囲全体の照合（検証）を通過した候補のみ採用」で、
    検証できない浅い一致は採用しない（誤結合より単純連結＋通知の方が安全）
- 一覧・月フィルタ・合計表示。並び順は「追加順」（取り込んだ順の新しい順・既定）と
  「日付順」（領収書日付の新しい順。日付なしは末尾、同日中は追加順）を
  切り替えられ、選択は記録されて次回も引き継がれる。一覧の各行は種別で色分けされる:
  勘定科目が交通費の行はティールの色帯＋「🚃 交通費」バッジ、
  宿泊費の行はローズの色帯＋「🏨 宿泊費」バッジ、
  それ以外はアンバーの色帯＋「🧾 購入・立替」バッジ（取込タブの大項目と対応。
  いずれもテーマの紫と被らない色にして見分けやすくする）
- 「取込」「一覧」「設定」タブは、画面を更新（リロード）しても直前まで開いていた
  タブのまま表示される（localStorageに記録）
- **PDF領収書の取込**: 楽天トラベル等が発行するPDF領収書を、購入費・交通費（🎫）・
  宿泊費・写真の後付けのどの入口からでも選択できる。**pdf.js**（同梱・
  `docs/vendor/pdfjs/`、PDFが選ばれたときだけ遅延読み込み）で画像化し、
  写真と同じレシートOCR→フォーム反映→証憑保存（PNG・幅約1800px）のフローに乗せる。
  デジタル文書のためトリミング編集はスキップ。複数ページは縦に連結（iOSのキャンバス
  面積上限に収まるよう最大3ページ）し、縦長になった場合は分割してOCRへ送る。
  ※pdf.jsは本プロジェクト初のサードパーティ依存（PDFレンダラーの自前実装は
  非現実的なため例外とした。hub DECISIONS.md参照）
- **写真の後付け**: 証憑画像の無い既存明細（手入力等）の編集画面に
  「📷 写真を追加して解析」を表示。交通費はスクショを直接解析
  （複数枚選択時は重なり連結して1枚に）、それ以外はトリミングエディタを経由して解析し、
  結果を編集中のフォームへ反映（依頼内容・按分は保持、空の項目は据え置き）。
  保存を押すと証憑画像として明細に添付される（既存画像の差し替えはしない）
- **画像から再解析**: 一覧から開いた既存明細（証憑画像あり）の編集画面に
  「🔁 画像から再解析」ボタンを表示。保存済みの証憑画像でOCRをやり直し、
  フォームの各項目を上書きする（保存を押すまで明細には反映されない）。
  勘定科目が交通費なら経路プロンプト（縦長の連結スクショは分割して送信）、
  それ以外はレシートプロンプトで解析。依頼内容・用途と按分人数は利用者の記入のため
  保持し、読み取れなかった項目（空・0）も現在の値を据え置く。
  宿泊費の明細はOCRの科目判定では科目を変えない（入口で種別確定のため）
- 「明細を編集」「内容を確認・修正」の画面も、開いたまま画面が更新されても復元される。
  編集中の既存明細はIDを記録して再度開き直す。OCR解析直後・手入力途中の未保存内容は
  下書きとしてlocalStorage（項目）＋IndexedDB（証憑画像）に保持し、保存またはキャンセル
  で消える。iOSがバックグラウンドでタブを再読み込みしてしまう場合でも、OCR解析の
  API呼び出し分の結果や入力内容が無駄にならないようにするための対応
- **ZIPエクスポート（CSV＋証憑画像）**: CSVの「元ファイル名」列とZIP内の画像ファイル
  （`日付_支払先_金額_ID.jpg`）が1対1で紐づく。経費精算の証憑提出にそのまま使える
- CSVのみのエクスポートも可能（UTF-8 BOM付き8列、ExpenseVBAの明細シートと同一列順）
- 一覧の行をタップすると編集画面が開く（証憑画像・取込日時は維持）。会議名・案件を
  記録する備考欄あり。「申請」「証憑画像を保存」「この明細を削除」もすべて編集画面に集約
  （一覧上にはボタンを置かず、誤タップを防いでいる）
- **「申請」ボタン**: Web共有API（`navigator.share`）で証憑画像＋明細をメール等に共有できる。
  対応環境ではメールアプリの作成画面が証憑画像添付済み・本文入力済みの状態で開く
  （宛先は仕様上事前入力できないため利用者が入力する）。非対応環境では`mailto:`にフォールバック
  - 共有先アプリが本文・添付を正しく展開しないことがある（例: Windows版新しいOutlookで
    添付が空ファイルになる・本文/件名が反映されない事例を確認済み。受信側アプリの実装の
    問題でこちら側では制御不可）ため、共有と同時に本文を自動でクリップボードにもコピーする
  - 各明細に「証憑画像を保存」ボタンを用意し、共有がうまくいかない場合は証憑画像を直接
    ダウンロードして手動添付できるようにしている
- **按分計算**（編集画面）: 複数人での飲み会等を一旦立て替えた後、1人あたりの請求額を
  出す機能。金額と人数から`Math.floor(金額 / 人数)`で1人分を計算し、端数は立て替えた
  本人が持つ前提（他の人には請求しない）。人数の既定値は1で、1＝按分なしの通常登録、
  2以上＝按分として記録、0は無効。2以上のときだけ一覧・編集画面・CSV
  （末尾に「按分人数」「1人あたり請求額」の2列を追加）・「申請」の共有本文に反映される。
  明細の金額欄自体はレシート記載の合計金額のまま変更しない
- APIキー・明細・画像はすべて端末内にのみ保存（サーバーに一切送信されない。送信先はOpenAIのみ）
- 設定タブに「OpenAIの利用状況を見る」リンクあり。OpenAIはAPIキー経由でクレジット残高を
  取得する公式APIを提供していないため（非公式・非推奨のエンドポイントのみ）、アプリ内に
  数値を表示する実装はせず、公式の利用状況ページを別タブで開く方式にとどめている
- iOS Safariの「ホーム画面に追加」でアプリのように起動（PWA manifest対応）

## ローカル開発

### backend

```bash
docker run -d -e POSTGRES_PASSWORD=dev -p 5432:5432 postgres:16

cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

DATABASE_URL=postgresql://postgres:dev@localhost/expenseweb alembic upgrade head
DATABASE_URL=postgresql://postgres:dev@localhost/expenseweb \
  ADMIN_USERNAME=admin ADMIN_PASSWORD=devpass \
  OPENAI_API_KEY=sk-xxxx OPENAI_MODEL=gpt-5.4 \
  uvicorn app.main:app --reload
```

テスト（OpenAI呼び出しはモック、実課金なし）:

```bash
cd backend
pytest
```

### frontend

```bash
cd frontend
npm install
API_BASE_URL=http://localhost:8000/api/v1 \
  ADMIN_USERNAME=admin ADMIN_PASSWORD=devpass \
  npm run dev
```

ブラウザで `http://localhost:3000` を開き、Basic認証（admin / devpass）でログインする。

## Railwayへのデプロイ手順

`docs/RAILWAY_STACK_GUIDE.md`のRailwayセットアップ手順（3サービス構成）に準拠。

1. [railway.app](https://railway.app) で新規プロジェクト作成 → GitHubリポジトリ（`hayahide-cloud/Shiden-Expense`）を接続
2. サービスを3つ追加
   - **PostgreSQL**（Railwayテンプレート、自動作成）
   - **backend**（Root Directory: `backend`）
     - Build Command: `pip install -r requirements.txt && alembic upgrade head`
     - Start Command: `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
     - 環境変数: `DATABASE_URL=${{Postgres.DATABASE_URL}}`, `ADMIN_USERNAME`, `ADMIN_PASSWORD`, `OPENAI_API_KEY`, `OPENAI_MODEL=gpt-5.4`
   - **frontend**（Root Directory: `frontend`）
     - Build Command: `npm ci && npm run build`
     - Start Command: `npm start`
     - 環境変数: `API_BASE_URL=http://${{backend.RAILWAY_PRIVATE_DOMAIN}}/api/v1`, `ADMIN_USERNAME`（backendと同じ）, `ADMIN_PASSWORD`（backendと同じ）
3. frontendサービス → Settings → Networking → 「Generate Domain」で公開URLを発行
4. 発行されたURLにスマホのSafariでアクセスし、Basic認証でログイン後、「ホーム画面に追加」でアプリのように起動できる

## 使い方

1. `/capture` で領収書写真を撮影 →「解析する」→ 抽出結果を確認・修正 →「保存」
2. `/receipts` で一覧確認・月フィルタ・CSVエクスポート
3. エクスポートしたCSVを、ExpenseVBAの明細シートへコピー＆ペーストで取り込む（自動連携は未対応、手動ブリッジ）

## セキュリティ

- 単一オーナー専用の非公開ツール。全ページ・全APIをBasic認証で保護
- 領収書画像はサーバーに保存しない（解析後は破棄。ストレージコスト増加を避けるため）
